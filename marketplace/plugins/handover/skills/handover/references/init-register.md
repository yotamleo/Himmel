# `/handover init`, `/handover register`, `/handover repos` — Full Specs

Load when running any of these commands, or when troubleshooting registry state.

## Pre-flight Validation (shared)

The `bucket_name` collision check below is run by all three write paths
(`/handover init`, `/handover register`, `/handover repos add`) before they
write anything. Each path's pre-check list references this block and notes
only WHEN it runs relative to that path's writes.

**`bucket_name` collision (refuse, rc=2).** Resolve the effective
`bucket_name` (explicit `--bucket-name`, else slugified `basename(path)` —
same rule as the `bucket_name` row in the `init` Inputs table). If it already
maps to a **different** registered repo, refuse: `bucket_name '<bucket>'
already used by repo '<other-name>'; pass --bucket-name=<unique> to
disambiguate.` Runs before any filesystem/registry write so a collision
leaves no partial disk state.

## `/handover init`

Bootstraps a **new** repo (no existing handover state on disk).

### Inputs (interactive prompt via `AskUserQuestion` or text)

| Field | Default | Validation |
|---|---|---|
| `name` | repo-root basename, slugged | non-empty, kebab-case, ≤30 chars, unique in registry |
| `path` | canonical `parent(git -C $cwd rev-parse --git-common-dir)` | abs path, exists, is a git repo |
| `user` | slugged `git -C <path> config user.name` (fallback `$USER`) | non-empty, kebab-case |
| `aliases` | `[name]` | optional, kebab-case list |
| `keywords` | `[]` | optional, free-text list |
| `branch_prefix` | `handover/` | optional, must end with `/`; used by worktree gate for `<branch_prefix><slug>`. Scopes **handover-mutation** branches (`new-epic`, `new-task`, `new-standalone`, `end-session`, `update-status`) — distinct from general feature-branch prefixes used by `/worktree.sh` (CLAUDE.md `type/slug` convention) |
| `jira_project` | none | optional, e.g. `HIMMEL` — enables Jira mirror on `new-epic` and `end-session` |
| `bucket_name` | slugified `basename(path)` | optional (HIMMEL-147); used by HIMMEL-129 bucket-layout resolver to decide which `<state-root>/<bucket>/` directory this repo's handover state lands in. **Collision check:** `/handover register` + `/handover repos add` refuse when two repos resolve to the same `bucket_name`; pass `--bucket-name=<unique>` to disambiguate. Existing entries auto-backfill from `basename(path)` on first read. |

### Pre-checks (refuse on failure)

1. `<path>` is a git repo (`git -C <path> rev-parse --is-inside-work-tree`).
2. `<path>` is canonicalised; same value not already in registry.
3. `<name>` not already in registry.
4. **`bucket_name` collision (refuse, rc=2).** See [Pre-flight Validation (shared)](#pre-flight-validation-shared). Runs before any filesystem write.
5. `<path>/handovers/<user>/` is **empty of `.md` files** (`.gitkeep` allowed). If non-empty, refuse and tell user to run `/handover register` instead.

### Bootstrap steps

1. Create dirs:
   - `<path>/handovers/<user>/`
   - `<path>/handovers/<user>/epics/`
   - `<path>/handovers/<user>/standalones/`
   - `<path>/handovers/<user>/_templates/`
2. Copy all files from `${CLAUDE_PLUGIN_ROOT}/templates/*.md` → `<path>/handovers/<user>/_templates/`. Each template carries `template_version: 1` frontmatter (added at plugin-build time; copy verbatim).
3. Resolve the plugin templates source:
   - If `$CLAUDE_PLUGIN_ROOT` is **unset or empty** → refuse: `CLAUDE_PLUGIN_ROOT is not set; reinstall the handover plugin or run from a Claude Code session that exposes the plugin env.`
   - Else if `$CLAUDE_PLUGIN_ROOT/templates/` **does not exist on disk** → refuse: `Templates missing under <expanded $CLAUDE_PLUGIN_ROOT>/templates/; reinstall the handover plugin.`
   - Do not attempt cache discovery — discovery is fragile across Git Bash, PowerShell, and WSL on Windows and silently picks a stale plugin version when multiple are cached.
4. Write `<path>/handovers/<user>/status.md`:

   ```markdown
   # Handover Status — <user> @ <name>

   > Auto-regenerated by Claude. Do not edit manually.
   > Last updated: YYYY-MM-DD

   ## Summary

   0 epic(s) | 0 standalone(s) | 0 open bugs | 0 blocked

   ## Epics

   _none yet_

   ## Standalones

   _none yet_
   ```

5. Write `<path>/handovers/<user>/backlog.md`:

   ```markdown
   # Backlog — <user> @ <name>

   _Unprioritised future work. Promote to epic/standalone when ready._
   ```

6. Write `<path>/handovers/<user>/counter.md`:

   ```markdown
   # Counter — <user> @ <name>

   Next: 1
   ```

7. **Atomic registry write** (see `routing.md`). Add entry:

   ```json
   "<name>": {
     "path": "<canonical path>",
     "user": "<user>",
     "aliases": ["<name>", ...],
     "keywords": [...],
     "branch_prefix": "handover/",  // omit to accept default
     "jira_project": "<KEY>"        // omit if not set; omitting disables Jira mirror
   }
   ```

8. Confirm: `Initialised repo '<name>' at '<path>/handovers/<user>/'. Run /handover repos list to verify.`

### Idempotency

Re-running `init` for the same `<name>` or `<path>` is refused. The check happens before any filesystem writes.

If a partial init left files behind (process killed mid-bootstrap), the next run refuses on step 5 of pre-checks (non-empty handover dir). Operator manually cleans up or uses `register` to adopt.

## `/handover register`

Adopts **existing** handover state in a repo. Idempotent — re-run is always safe.

### Inputs

Same as `init`, but `name`, `aliases`, `keywords`, `jira_project` may default to the values in registry if a partial registration exists.

### Pre-checks (warn but proceed; refuse only on hard errors)

1. `<path>` is a git repo.
2. `<path>/handovers/<user>/` exists.
3. `<path>/handovers/<user>/_templates/` exists OR plugin defaults exist for seeding.
4. **`bucket_name` collision (refuse, rc=2).** See [Pre-flight Validation (shared)](#pre-flight-validation-shared). Runs **before** the adoption-step `counter.md` write (adoption step 4) so a collision leaves no partial disk state.

### Adoption steps

1. **Scan for IDs:**
   - `<state-root>/epics/#N-*/` → collect N
   - `<state-root>/epics/*/tasks/#N-*/` → collect N
   - `<state-root>/standalones/#N-*/` → collect N

2. **Duplicate detection:**
   - If any N appears in more than one path, output:

     ```
     WARN: duplicate ID #<N> at <path1> and <path2>
     ```

     Sort by N asc. Trailing line: `Found <K> collision(s). Counter set to <Next>; collisions are advisory only (no auto-fix).`
   - Do **not** rename or delete. Continue.

3. **Derive Next:**
   - `derived = max(all unique IDs) + 1` (or `1` if empty).
   - If `<state-root>/counter.md` exists with `Next: K`:
     - `K > derived` → preserve K, warn: `counter.md has Next: <K> > filesystem max + 1 (<derived>); preserving higher value.`
     - `K <= derived` → use `derived`.
   - Else use `derived`.

4. **Write `counter.md`** with the chosen `Next`. Overwrite is safe (idempotent).

5. **Template version stamp injection** (one-shot, idempotent):
   - For each file in `<state-root>/_templates/*.md`:
     - If no `template_version:` line in frontmatter, insert `template_version: 1` after the opening `---` (or prepend frontmatter if file has none).
   - If `<state-root>/_templates/` is missing, seed from plugin defaults (same as `init` step 2-3).

6. **Regenerate `status.md`** via `update-status` logic.

7. **Atomic registry write** with the registration entry.

8. Confirm: `Registered repo '<name>' at '<path>/handovers/<user>/'. Next ID: <K>. <duplicate-summary if any>.`

### Re-run safety

Re-running `register` against the same path:

- Re-scans for IDs and duplicates (always fresh).
- Preserves higher `counter.md` value if present.
- Stamp injection is idempotent (skips files already stamped).
- Registry entry is overwritten (last-write-wins), but `name` collision against a different `path` is refused.

## `/handover repos <subcommand>`

Registry-only operations. Never touch repo files.

### `list`

Output as table:

```
NAME      PATH                                                              USER     ALIASES                       JIRA
himmel    c:/users/<user>/documents/github/himmel                             yotam    himmel,internal                HIMMEL
luna      c:/users/<user>/documents/github/luna                               yotam    luna,vault,second-brain        —
```

If empty: `No repos registered. Run /handover init or /handover register.`

### `add <name> <path>`

Non-interactive register. Pre-checks:

1. `<path>` canonicalised.
2. `<path>/handovers/<user>/_templates/` exists AND `<state-root>/counter.md` exists.
   - If both missing: refuse with `Path not bootstrapped. Run /handover init <name> for new state, or /handover register for existing state.`
3. `<name>` not already registered.
4. `<path>` not already registered under a different name.
5. **`bucket_name` collision (refuse, rc=2).** See [Pre-flight Validation (shared)](#pre-flight-validation-shared). Registry-only path (no disk write).

If checks pass, write registry entry (atomic).

### `remove <name>`

Refuse if not in registry. Otherwise atomic-remove. Does not delete any files on disk.

### `where <name>`

Print:

```
name:   <name>
path:   <canonical path>
state:  <path>/handovers/<user>/
user:   <user>
jira:   <KEY or —>
```

## Registry shape — v2 (additions)

v2 adds four optional fields to each repo entry. All have safe defaults; absence means "use built-in default".

```jsonc
{
  "repos": {
    "<name>": {
      // ...v1 fields unchanged: path, user, aliases, keywords, branch_prefix, jira_project...

      // v2 additions:
      "bucket_vocab": "time-horizon",        // "time-horizon" | "kanban"; default "time-horizon"
      "buckets_custom": null,                // optional 4-element array overriding bucket names
      "defaults": {                          // per-question save-defaults; absence = ask
        "new_epic.jira_autocreate": true,
        "new_task.jira_autocreate": true,
        "new_standalone.jira_autocreate": true,
        "new_*.bucket": "next",
        "slug_style": "prompt-on-ambiguous",
        "worktree.reuse_on_match": "ask"
      },
      "stale_thresholds_days": {             // tech-debt.md tiers; defaults shown
        "warming": 30,
        "stale": 60,
        "zombie": 90
      },

      // HIMMEL-307 — host-repo-only addition (set on the state-root host, e.g. yotam_docs):
      "source_buckets_extra": ["luna-medic"] // optional; extra source-bucket names beyond the 4 built-ins. Absent/empty ⇒ 4-set.
    }
  }
}
```

### Defaults key reference

| Key | Type | Values | Meaning |
|---|---|---|---|
| `new_epic.jira_autocreate` | bool | true/false | skip "create Jira Epic?" prompt |
| `new_task.jira_autocreate` | bool | true/false | skip "create Jira Task?" prompt |
| `new_standalone.jira_autocreate` | bool | true/false | skip "create Jira Story?" prompt |
| `new_*.bucket` | string | one of bucket names | default bucket on new items (skip "which bucket?" prompt) |
| `slug_style` | string | `prompt-on-ambiguous` \| `strict-hyphen-only` | how to handle ambiguous slug derivation |
| `worktree.reuse_on_match` | string | `ask` \| `reuse` \| `new` | when target branch exists and merge state unclear |

### `source_buckets_extra` (HIMMEL-307)

Optional **host-repo-only** field — set it on the registry entry whose
`<state-root>` actually hosts the per-source-repo bucket directories (for this
operator, `yotam_docs`). It extends the **recognized source-bucket set** (see
SKILL.md "Bucket Resolution") beyond the four built-ins
(`himmel`/`luna`/`luna_brain`/`cross`).

| Property | Value |
|---|---|
| Type | array of strings (kebab-case, ≤30 chars each) |
| Default | absent ⇒ `[]` ⇒ recognized set = the four built-ins (byte-identical to pre-HIMMEL-307) |
| Where | the state-root **host** repo entry only (not every repo) |
| Routing | **explicit-only** — extra buckets get no Jira-prefix auto-route; items land there via the `new-epic`/`new-standalone` source-bucket prompt or `/handover bucket <id> <extra>` |

**Distinct from two same-sounding fields — do not conflate:**

- `buckets_custom` — renames the **time-horizon** vocab (`now/next/later/someday`). A *different axis*; it cannot add a source bucket.
- `bucket_name` — a single per-source-repo label used by the prefix rule to pick which built-in bucket that repo's Jira-keyed work lands in. `source_buckets_extra` instead declares *additional* destination buckets on the host.

Validation: each name must be kebab-case, ≤30 chars, and **disjoint** from the
four built-in source-bucket names AND from **every** reserved time-horizon name
the `/handover bucket` dispatcher and the frontmatter reader can accept — i.e.
both vocab sets `{now, next, later, someday, wip, next-up, backlog, icebox}`
(buckets.md "accept either name" means the kanban alts are accepted even when
`time-horizon` is the active vocab) **plus** any `buckets_custom` value in any
registered repo. Reject collisions so `/handover bucket` axis dispatch stays
unambiguous. This collision check runs whenever `source_buckets_extra` is
written to the host-repo entry — same write-path discipline (atomic write,
refuse-on-collision) as the `bucket_name` collision check above.

### Init/register flow changes

`/handover init`:
1. After the existing prompts (name, path, user, aliases), additionally prompt for `bucket_vocab` (default `time-horizon`).
2. Write the v2 fields to the registry entry. `defaults: {}` starts empty (skill will offer save-default opt-in on each first-time ask).

`/handover register`:
1. If the registry entry exists pre-v2 (no `bucket_vocab` field), inject defaults with `bucket_vocab: "time-horizon"`, `defaults: {}`, `stale_thresholds_days: {30, 60, 90}`. Never overwrite an existing v2 value.

`source_buckets_extra` (HIMMEL-307) is **not** prompted by `init`/`register` —
there is no dedicated subcommand to set it yet (a follow-up may add one). It is
set explicitly on the state-root host repo via a one-time registry edit.
Whenever it is written, the write path MUST run the disjointness validation
above before persisting; a collision refuses the write (rc=2) and leaves the
registry untouched, mirroring the `bucket_name` collision check. Absent the
field, every flow is byte-identical to pre-HIMMEL-307.
